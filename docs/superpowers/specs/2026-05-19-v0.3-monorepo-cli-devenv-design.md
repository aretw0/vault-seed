# Design: v0.3 — Monorepo, CLI Distribuível e Ambiente de Desenvolvimento

**Data:** 2026-05-19  
**Status:** Aprovado  
**Escopo:** Minor release — vault-seed v0.3.x

## Resumo

Transformar o vault-seed numa base monorepo incremental que hospeda o primeiro
pacote distribuível (`@dgk/cli`), fornece um devcontainer portátil com suporte
a PT-BR, e moderniza o setup local para ser multiplataforma. O vault template
permanece intacto na raiz; a infraestrutura de desenvolvimento cresce ao redor
dele sem quebrar usuários existentes.

---

## 1. Estrutura do Repositório

O vault permanece na raiz. Novos diretórios crescem ao redor dele:

```
vault-seed/
├── packages/                        ← workspace de pacotes publicáveis
│   └── cli/                         ← @dgk/cli — primeiro distribuível
│       ├── src/
│       │   ├── index.js
│       │   └── commands/
│       │       ├── validate.js
│       │       ├── lint.js
│       │       ├── setup.js
│       │       ├── release.js
│       │       └── check.js
│       ├── package.json
│       └── README.md
├── .devcontainer/                   ← ambiente portátil (vai com o usuário)
│   ├── devcontainer.json
│   └── post-create.sh
├── .changeset/                      ← versioning do CLI (não vai com usuário)
│   └── config.json
├── pnpm-workspace.yaml              ← workspace root (não vai com usuário)
├── package.template.json            ← package.json do vault do usuário
├── pnpm-lock.template.yaml          ← lock file do vault do usuário
├── scripts/                         ← mantido; CLI wrappeia estes
├── docs/                            ← mantido
├── [pastas do vault 00–99]          ← vault intacto
├── package.json                     ← workspace root (private: true)
├── VERSION                          ← versão do vault (standard-version)
└── CHANGELOG.md                     ← changelog do vault (standard-version)
```

**Decisões chave:**
- `package.json` raiz torna-se `"private": true` — workspace root do template
- `package.template.json` é o `package.json` que o usuário recebe, com `@dgk/cli` como devDependency
- `pnpm-lock.template.yaml` é gerado rodando `pnpm install` com `package.template.json` e commitado manualmente a cada mudança de dependência do usuário
- `packages/`, `.changeset/`, `pnpm-workspace.yaml` não chegam ao usuário final

## 2. CLI Núcleo — `@dgk/cli`

### Identidade do pacote

```json
{
  "name": "@dgk/cli",
  "version": "0.1.0",
  "bin": { "dgk": "./src/index.js" },
  "type": "module",
  "publishConfig": { "access": "public" }
}
```

Instalável via `npm install -g @dgk/cli` ou `npx @dgk/cli`. Binário: `dgk`.

### Comandos núcleo

| Comando | Comportamento | Origem |
|---|---|---|
| `dgk validate` | lint + testes + onboarding + smoke | `pnpm run validate` |
| `dgk lint` | markdownlint nas pastas do vault | `pnpm run lint` |
| `dgk setup` | configura git, python, node | `scripts/setup.sh` |
| `dgk release` | bump de versão do vault | `standard-version` |
| `dgk check` | saúde rápida (links, estrutura PARA) | `validate_onboarding.js` |

### Implementação

- Parse de argumentos via `node:util parseArgs` — zero dependências de runtime publicadas
- Lógica de cada comando delega para os scripts existentes em `scripts/`
- Nenhuma lógica nova nessa minor — o CLI é invólucro com interface consistente

## 3. Fronteira do Template

O `initialize.yml` usa `files_to_rename` (executado primeiro via `mv`) e
`files_to_remove` (executado depois). A ordem importa: renomear antes de
remover significa que o destino do rename não deve aparecer em `files_to_remove`.

### `initialize.yml` atualizado

```yaml
files_to_rename: >
  README.template.md:README.md
  CONTRIBUTING.template.md:CONTRIBUTING.md
  AGENTS.template.md:AGENTS.md
  package.template.json:package.json
  pnpm-lock.template.yaml:pnpm-lock.yaml

files_to_remove: >
  docs
  AGENTS.template.md README.template.md CONTRIBUTING.template.md
  packages .changeset pnpm-workspace.yaml
  .github/workflows/template-ci.yml
  .github/workflows/prepare-release-pr.yml
  .github/workflows/release.yml
  .github/workflows/security-audit.yml
  .github/workflows/sync-develop-with-main.yml
```

**Regra:** `package.json` não entra em `files_to_remove` — o `mv` de
`package.template.json:package.json` já sobrescreve o workspace `package.json`.

### O que o usuário recebe

| Arquivo/pasta | Status |
|---|---|
| `.devcontainer/` | ✓ mantido — ambiente portátil |
| `package.json` | ✓ substituído por `package.template.json` |
| `pnpm-lock.yaml` | ✓ substituído por `pnpm-lock.template.yaml` |
| `packages/` | ✗ removido — instala CLI via npm |
| `.changeset/` | ✗ removido — dev tooling do template |
| `pnpm-workspace.yaml` | ✗ removido — sem workspace local |

## 5. Setup Local — Multiplataforma

### Ferramentas por camada

| Camada | Ferramenta preferida | Fallback |
|---|---|---|
| Node version | `fnm` (Windows/macOS/Linux) | `nvm` (Linux/macOS) |
| Node packages | `pnpm` via Corepack | — |
| Python tools | `uv` (Windows/macOS/Linux) | `pipx` |

### `setup_node.sh` revisado

```bash
#!/bin/bash
set -e
echo "Configurando ambiente Node.js..."

if command -v fnm &>/dev/null; then
  eval "$(fnm env --use-on-cd)"
  fnm use --install-if-missing
  echo "Node.js $(node -v) ativado via fnm."
elif [ -s "$HOME/.nvm/nvm.sh" ]; then
  echo "[AVISO] nvm encontrado. Recomendamos migrar para fnm (cross-platform)."
  echo "Instale fnm: https://github.com/Schniz/fnm#installation"
  \. "$HOME/.nvm/nvm.sh"
  nvm use --install-if-missing
else
  echo "[ERRO] Nenhum gerenciador de versão Node encontrado."
  echo "Instale o fnm:"
  echo "  Windows : winget install Schniz.fnm"
  echo "  macOS   : brew install fnm"
  echo "  Linux   : curl -fsSL https://fnm.vercel.app/install | bash"
  exit 1
fi

corepack enable
corepack prepare --activate
[ ! -d "node_modules" ] && pnpm install --frozen-lockfile
echo "Configuração do ambiente Node.js concluída."
```

### `setup_python.sh` revisado

```bash
#!/bin/bash
set -e
echo "Configurando ambiente Python..."

if ! command -v python3 &>/dev/null; then
  echo "[ERRO] Python 3 não encontrado."
  exit 1
fi
echo "Python $(python3 --version) encontrado."

if command -v uv &>/dev/null; then
  uv tool install git-filter-repo
else
  echo "[AVISO] uv não encontrado. Usando pipx como fallback."
  echo "Recomendamos instalar uv: https://docs.astral.sh/uv/getting-started/installation/"
  if ! command -v pipx &>/dev/null; then
    python3 -m pip install --user pipx
    python3 -m pipx ensurepath
  fi
  pipx install git-filter-repo
fi

echo "Configuração do ambiente Python concluída."
```

### `setup_git.sh` revisado

```bash
#!/bin/bash
set -e
echo "Configurando Git..."
git config commit.template .gitmessage
git config core.quotepath false
git config i18n.commitEncoding UTF-8
git config i18n.logOutputEncoding UTF-8
```

## 7. CI — O Que Muda

**`template-ci.yml`** — adiciona validação do workspace CLI:
```yaml
- name: Validate CLI package
  run: pnpm --filter @dgk/cli test
```

**Novo `publish-cli.yml`** — publica `@dgk/cli` no npm:
```yaml
on:
  push:
    tags: ['@dgk/cli@*']
jobs:
  publish:
    steps:
      - uses: actions/checkout@<SHA-pinado>  # pinado por SHA como todos os outros workflows
      - uses: ./.github/actions/setup
      - run: pnpm --filter @dgk/cli build
      - run: pnpm changeset publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

**Não mudam:** `ci.yml`, `release.yml`, `prepare-release-pr.yml`, `security-audit.yml`.
